传统助手 vs Claude Code (Agent)

维度 传统聊天助手 Claude Code (Agent)

上下文管理三层策略

LEVEL 1  基础
/clear 任务切换时完全重置 /compact 手动压缩，可指定保留焦点
LEVEL 2  中级
CLAUDE.md <200 行 精简项目指令，避免超载 Auto Memory 跨 session 持久化笔记 /btw 侧边快问，不写入历史
LEVEL 3  高级
Subagent delegation 重型任务外包到独立上下文 Skills lazy-load 按需加载，不占常驻空间 Pipeline state JSON 状态持久化阶段信息 Multi-session 跨会话拆分长任务

核心原则：主动管理上下文 = 持续的高质量输出

Hook — 事件驱动的自动化

Shell 脚本在特定事件触发 — 确定性执行，不是建议

类型 触发时机 能力
PreToolUse 工具执行前 阻止 + 验证 + 修改参数
PostToolUse 执行后 格式化 + 检查 + 推进流水线
Stop 即将停止时 阻止停止 + 注入提醒
PreCompact 压缩前 设标记 + 保存状态

Source: code.claude.com/docs/en/hooks-guide

Hook 实战: 实验流水线

dev ──→ skill-update ──→ experiment ──→ monitoring ──→ analysis ──→ commit ──→ dev (cycle) █
  • Stop hook 检测到 10+ 文件编辑 → 提醒运行 /writing-skills
  • PostToolUse 检测 Skill 完成 → 自动推进到下一阶段
  • Stop hook 检测实验完成 → 提醒 /slides-dispatch
  • 手动控制: pipeline-ctl.sh {status|set|skip|reset}

Hook 实战: 安全护栏

  • slides-guard.sh — PreToolUse
    拦截对重型 skill 的直接调用，建议使用 dispatcher
    [slides-guard] Direct use of /agent-slides would fill main context.
    Use /slides-dispatch instead — delegates to subagent automatically.
  • console.log audit — Stop hook
    会话结束前审计所有修改文件，阻止残留 console.log 提交
  • git push review — PreToolUse
    推送前在编辑器中打开 diff 供人工审查，确认后再执行

Skills 的可扩展性

  • writing-skills — 用 TDD 方式创建新 skill
    先写期望行为，再迭代 skill 内容直到满足测试
  • find-skills — 从开放生态发现并安装 skill
    $ npx skills # 浏览社区 skill 目录
  • "你可以构建自己的研究工作流 skill"
    任何重复性研究流程都可以 skill 化，一次编写，永久复用

Sub-agent — 独立上下文的专家

  • 独立 context window + system prompt + tools
    主 context 保持干净，重任务在一次性 context 里完成
  • 内置类型:
    Explore (Haiku, 快速) Plan General-purpose Custom
  • ⚠ 不能 spawn 其他 subagent — 防止无限嵌套与资源爆炸

Source: code.claude.com/docs/en/sub-agents

Subagent Delegation Pattern

Main Context (clean) Subagent Context (disposable) ──────────────────── ───────────────────────────── User request 读取 skill.md (~60KB) | 读取实验数据 收集需求 (~10 lines) 生成 HTML (~3000 lines) | 写入文件 Spawn Agent tool ──────────────> | <────────── 返回 summary (5 lines) 向用户报告文件路径
"重活在一次性 context 里做，结果在干净 context 里收"

Source: code.claude.com/docs/en/sub-agents

Before / After: 实验生命周期

传统工作方式
  • 手动追踪实验状态
    表格/笔记，容易遗漏
  • Context 在 session 结束后丢失
    下次要重新解释背景
  • 无自动提醒下一步
    靠记忆，易遗忘关键步骤
  • 结果散落各处
    slides / logs / notes 不同位置
Pipeline + Hooks 工作方式
  • Pipeline hooks 自动追踪阶段
    state.json 持久化，随时查询
  • Memory 跨 session 保留知识
    CLAUDE.md + skills 永不遗忘
  • Stop hook 自动提醒下一步
    确定性执行，不依赖记忆
  • 自动生成 slides + changelog
    实验结束即产出，统一归档

CLAUDE.md 最佳实践

DO 写进去
  • 构建命令 (build / test / lint)
  • 代码风格偏好 (const/let, types)
  • 测试指南 (框架、覆盖率要求)
  • 架构决策 (目录结构、分层规则)
DON'T 不要写
  • Claude 能自行推断的内容
  • 标准行业惯例 (众所周知)
  • 详细 API 文档 (用 Skills 代替)
  • 频繁变化的配置 (易失效)
Hooks > CLAUDE.md (100% 执行，不可绕过)   |   Skills > CLAUDE.md (偶尔需要的知识，按需加载)

Source: code.claude.com/docs/en/best-practices

9 条 Coding Agent 原则

少即是多
不装插件，基础 CLI 已足够
Context is Everything
研究与实现阶段分离，指令精确
利用讨好倾向
用中立 Prompt，防止 Agent 编造
避免填空
在 CLAUDE.md 设规则，防止乱猜
定义终点
测试驱动 / CONTRACT.md 验收标准
CLAUDE.md 作逻辑目录
写 IF-ELSE 树指向规则文件，而非百科全书
Rules vs Skills
Rules = 偏好；Skills = 执行菜谱
定期清理
让 Agent 自我整理，删除矛盾规则
别追新工具
官方集成的才是真正有用的

Source: @systematicls on X — "Coding Agent Best Practices"

Boris Cherny 10 Tips (1–5)

  • 1. git worktree 多会话并行 — 同时运行多个 Claude 实例，互不干扰
  • 2. 先 Plan Mode 再实现 — 一个 Claude 写计划，另一个 Claude 审计划
  • 3. 每次纠正后更新 CLAUDE.md — 把规则写进去，防止同类错误复现
  • 4. 日常操作做成 Skills — 每天用 >1 次的操作变成自定义 slash command
  • 5. Slack MCP 让 Claude 独立修 bug — 粘贴 bug thread，让 Claude 自主查看 CI 日志

Source: @bcherny (Boris Cherny) Twitter thread, 2026-01-31

Boris Cherny 10 Tips (6–10)

  • 6. Claude 做 reviewer 挑战代码 — 让 Claude 扮演审查者，要求重写更优雅的实现
  • 7. Ghostty 终端 + 语音输入 — 更好的渲染 + 语音输入让 Prompt 更详细
  • 8. "use subagents" 投入更多计算 — 附加关键词，主 context 保持干净
  • 9. 直接用 BigQuery 等 CLI 做分析 — 无需手写 SQL，Claude 直接操作数据库实时分析
  • 10. 生成 HTML 演示文稿学习代码 — 让 Claude 生成可视化解释，ASCII 架构图辅助理解

Source: @bcherny (Boris Cherny) Twitter thread, 2026-01-31

常见失败模式

失败模式 问题 解决方案
厨房水槽 Session 把无关任务混入同一个会话，Context 污染 切换任务前先 /clear
反复纠正 失败尝试堆积，Context 里全是错误历史 纠正 2 次后 /clear，重新给干净 Context
CLAUDE.md 过长 内容太多被忽略，规则矛盾无法执行 无情删减，改用 IF-ELSE 逻辑目录 + Skills
信任验证差距 边界 case 未被覆盖，Agent 悄悄走错路 始终提供可验证的完成标准 (CONTRACT.md)
无限探索 Context 耗尽前仍在漫无目的地探索 缩小任务范围，或用 subagent 隔离探索
